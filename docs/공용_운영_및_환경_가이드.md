# 공용 운영·공동개발·데이터 이전 가이드

학부 홈페이지를 **여러 명이 함께 쓰거나**, **데이터 서버를 교내로 이전**할 때 참고하는 방법입니다.

---

## 1. 공용으로 쓸 때 (역할 분리)

### 원칙

- **코드·설정 템플릿**은 저장소에만 둡니다.
- **실제 비밀값(URL, API 키)** 은 각자 환경에만 넣고, 저장소에는 올리지 않습니다.

### 환경별로 필요한 것

| 역할 | 필요한 설정 | 비고 |
|------|-------------|------|
| **로컬 개발자 A** | 본인 PC에 `.env` (Supabase URL·anon key) | 로컬 Supabase 또는 팀 공용 Supabase 프로젝트 연결 |
| **로컬 개발자 B** | 동일하게 `.env` | 같은 Supabase 쓰면 같은 값, 따로 쓰면 다른 값 |
| **배포(운영)** | Cloudflare Pages 환경 변수 또는 GitHub Secrets | 운영용 Supabase 한 세트만 사용 |

### 공동개발 절차

1. **저장소 공유**  
   - GitHub 저장소를 팀/학부용으로 두고, 개발자는 `git clone` 후 `local` 등 개인 브랜치에서 작업.

2. **환경 변수 공유(비밀만)**  
   - Supabase를 **한 프로젝트**로 공용할 때: URL·anon key를 **비밀 채널**(비밀번호 관리 툴, 학부 보안 규정에 맞는 방법)로만 공유.  
   - 각자 `.env`에 동일 값 입력. `.env`는 절대 커밋하지 않음 (이미 `.gitignore`에 포함됨).

3. **신규 참여자**  
   - `README` 또는 이 문서 링크로 안내.  
   - 할 일: 저장소 클론 → `.env.example`을 복사해 `.env`로 저장 → 위에서 받은 URL·키 입력 → `npm install` → `npm run dev`.

4. **배포는 한 곳만**  
   - Cloudflare Pages(또는 사용 중인 호스트)는 **한 계정**에서만 관리하고, 환경 변수도 그쪽에만 넣어 두면 됩니다.

---

## 2. 데이터 서버를 교내로 이전할 때

현재 앱은 **Supabase**(PostgreSQL + Auth + Storage) 한 곳에만 연결됩니다.  
교내 서버로 옮기려면 아래 둘 중 하나를 선택하게 됩니다.

### 방법 A: Supabase를 교내에 설치 (Self‑hosted)

- Supabase는 오픈소스라 **직접 서버에 설치**할 수 있습니다.  
  [Supabase Self‑hosting 문서](https://supabase.com/docs/guides/self-hosting) 참고.
- 설치 후 나오는 **API URL**과 **anon key**만 바꾸면 됩니다.
- **앱 코드 변경**: 없음.  
  **변경하는 곳**:  
  - 로컬: `.env`의 `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`  
  - 배포: Cloudflare(또는 사용 중인 호스트) 환경 변수

### 방법 B: 교내 DB/API로 완전히 교체

- 교내 PostgreSQL + REST API(또는 다른 백엔드)로 바꾸는 경우입니다.
- **앱에서 바꿔야 할 곳**  
  - `src/lib/supabase.js`: Supabase 대신 교내 API URL·인증 방식에 맞는 클라이언트로 교체.  
  - 공지·갤러리·활동·로그인을 쓰는 훅/페이지:  
    - 데이터 소스를 Supabase 호출 → 교내 API 호출로만 바꾸면 됩니다. (테이블 구조를 맞추거나 어댑터 함수를 두면 유리합니다.)
- **문서**  
  - `docs/supabase-schema.sql`는 교내 PostgreSQL에 맞게 수정해 실행하면 됩니다.  
  - Auth는 교내 SSO·LDAP 등으로 바꾸면 됩니다.

### 이전 시 체크리스트

- [ ] 새 서버(또는 Supabase self‑hosted)에 DB 생성, 스키마 적용 (`docs/supabase-schema.sql` 참고)
- [ ] Storage 역할(갤러리·활동 대표 이미지)을 교내 스토리지 또는 새 API로 이전
- [ ] 인증(로그인)을 새 방식으로 연동
- [ ] `.env.example` 및 배포 환경 변수를 새 URL·키로 갱신
- [ ] 기존 Supabase 데이터 마이그레이션(필요 시 export → import)
- [ ] 배포 환경(Cloudflare 등)에서 환경 변수만 새 값으로 수정 후 재배포

---

## 3. 한 곳만 바꾸면 됩니다.

- **Supabase Cloud ↔ Supabase 교내 설치** 전환:  
  환경 변수만 바꾸면 됩니다.
- **다른 백엔드로 교체**:  
  `supabase.js`를 “API 클라이언트 진입점”으로 두고, 내부만 교체하면 됩니다.

앞으로도 **새 백엔드를 붙일 때는**  
- 가능하면 `supabase.js` 같은 **단일 진입점**을 두고,  
- URL·키는 **환경 변수**로만 주입하도록 하면,  
공용·이전 모두 관리가 쉽습니다.

## 4. 요약

| 목적 | 권장 방법 |
|------|-----------|
| **공용·공동개발** | 저장소는 공유, 비밀은 `.env`·비밀 채널로만 공유. 배포는 한 계정에서만 환경 변수 설정. |
| **교내 이전 (Supabase 유지)** | Supabase self‑hosted 설치 후, URL·키만 `.env`와 배포 환경에 반영. |
| **교내 이전 (다른 DB/API)** | `src/lib/supabase.js`를 교내 API 클라이언트로 교체하고, 스키마·Auth를 교내에 맞게 이전. |

추가로, **README**에 “실행 방법 → 환경 변수는 `.env.example` 참고, 자세한 공용·이전 절차는 `docs/공용_운영_및_환경_가이드.md` 참고” 한 줄을 넣어 두면 신규 참여자·운영자가 찾기 쉽습니다.

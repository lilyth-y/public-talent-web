# 인수인계·관리 가이드 (졸업 후 학교 인수인계용)

이 문서는 **개발·운영을 학교(학부)에 넘길 때** 필요한 관리 방법과 인수인계 체크리스트를 정리한 것입니다.  
콘텐츠만 수정하는 분, 배포·계정을 관리하는 분이 각각 참고할 수 있도록 구성했습니다.

---

## 1. 누가 무엇을 하나요?

| 역할 | 하는 일 | 참고 문서 |
|------|----------|------------|
| **콘텐츠 관리자** (행정팀·담당자) | 공지 올리기, 갤러리 사진·활동 글 등록·수정. **코드·배포는 건드리지 않음.** | [공지_등록_방법.md](공지_등록_방법.md), [갤러리_활동_등록_방법.md](갤러리_활동_등록_방법.md) |
| **배포·계정 관리자** (교내 담당자 또는 IT) | GitHub·Cloudflare·Supabase 계정, 비밀번호, 환경 변수. 코드 수정·배포가 필요할 때. | 이 문서 아래 "인수인계 시 넘겨야 할 것", "기술 정보" |

일상적으로는 **콘텐츠 관리자**가 `/admin` 로그인해서 공지·갤러리·활동만 관리하면 됩니다.  
디자인 변경, 새 메뉴 추가, 서버 이전 등은 **배포·계정 관리자** 또는 외주 개발자가 진행합니다.

---

## 2. 콘텐츠만 관리할 때 (매일·매주)

- **관리자 페이지**: 사이트 주소 + **`/admin`** (예: `https://학부사이트주소/admin`)
- **로그인**: Supabase에 등록된 이메일·비밀번호
- **할 수 있는 일**:
  - **공지사항**: 새 공지 추가, 수정, 삭제 → [공지_등록_방법.md](공지_등록_방법.md)
  - **갤러리**: 사진 업로드, 제목·설명, 순서 변경 → [갤러리_활동_등록_방법.md](갤러리_활동_등록_방법.md) 1번
  - **활동 게시글**: 행사 후기·활동 글 작성·수정·삭제 → [갤러리_활동_등록_방법.md](갤러리_활동_등록_방법.md) 2번

저장하면 **별도 배포 없이** 사이트에 바로 반영됩니다. (이미 배포된 사이트가 Supabase에 연결되어 있는 경우)

## 3. 인수인계 시 넘겨야 할 것

졸업(또는 담당 변경) 시 **배포·계정 관리자**가 받을 항목입니다.  
넘기는 쪽에서 목록을 채우고, 받는 쪽에서 접속·권한을 확인하면 됩니다.

### 3.1 계정·접속 정보 (보안에 맞게 전달)

| 항목 | 설명 | 받는 사람이 할 일 |
|------|------|-------------------|
| **GitHub** | 저장소 소유 계정 또는 팀. 코드가 여기 있습니다. | 저장소 접속 권한 또는 조직으로 이전, 비밀번호/2단계 인증 이전 |
| **Cloudflare** | 사이트가 배포되는 곳 (Pages). 도메인·SSL도 여기서 설정할 수 있음. | 계정 이전 또는 팀 초대, API 토큰·계정 ID 확인 |
| **Supabase** | 공지·갤러리·활동·로그인 데이터와 이미지 저장. | 프로젝트 소유권 이전 또는 팀 초대, 관리자 로그인 비밀번호 변경 권한 |

- **비밀번호·API 키**는 반드시 **비밀 채널**(학교 공식 비밀번호 관리, 담당자 간 직접 전달 등)로만 넘기고, 이메일 본문·공용 문서에 적지 마세요.

### 3.2 꼭 전달할 내용 정리

- **사이트 주소(URL)**: 실제 운영 주소
- **관리자 페이지**: `사이트주소/admin`
- **Supabase 로그인 계정**: 관리자로 쓰는 이메일 1개 (비밀번호는 별도 안전 전달)
- **GitHub 저장소 주소**: 예) `https://github.com/lilyth-y/public-talent-web`
- **배포 방식**: `main` 브랜치에 push하면 Cloudflare Pages가 자동 배포함
- **환경 변수**:  
  - 이 저장소는 GitHub Actions에서 `npm run build` 후 배포하므로, `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`를 **GitHub 저장소 Secrets**에 설정해야 합니다. (하위 호환: `SUPABASE_URL`, `SUPABASE_ANON_KEY`)  
  - Cloudflare Pages 환경 변수는 런타임 참고용으로 별도 유지해도 되지만, 현재 배포 파이프라인의 핵심은 GitHub Secrets입니다.

## 4. 기술 정보 (배포·개발 담당자용)

- **구조**: React + Vite 단일 페이지 앱(SPA). 백엔드는 Supabase(DB + 인증 + 스토리지)만 사용.
- **저장소**: GitHub. 기본 브랜치 `main`에 push 시 GitHub Actions로 빌드 후 Cloudflare Pages에 배포.
- **환경 변수**:  
  - 로컬: 프로젝트 루트 `.env` (저장소에 없음. `.env.example` 참고)  
  - 배포 빌드: GitHub 저장소 Secrets(`VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`; 하위 호환 `SUPABASE_URL`, `SUPABASE_ANON_KEY`)  
  - Cloudflare Pages 프로젝트 변수는 필요 시 별도 관리 가능
- **DB/스키마**: Supabase SQL Editor에서 `docs/supabase-schema.sql` 참고. 테이블: `notices`, `gallery_items`, `activity_posts` 등.
- **공용·교내 이전**: [공용_운영_및_환경_가이드.md](공용_운영_및_환경_가이드.md) 참고.

## 6. 문제가 났을 때 (점검 순서)

1. **관리자 페이지에 로그인이 안 돼요**  
   - Supabase에 해당 이메일 계정이 있는지, 비밀번호가 맞는지 확인.  
   - Supabase 대시보드 → Authentication → Users에서 비밀번호 재설정 가능.

2. **공지·갤러리·활동이 안 보여요 / 저장이 안 돼요**  
   - GitHub 저장소 Secrets에 `VITE_SUPABASE_URL`, `VITE_SUPABASE_ANON_KEY`(또는 `SUPABASE_URL`, `SUPABASE_ANON_KEY`)가 설정되어 있는지 먼저 확인.  
   - 값이 비어 있으면 배포 빌드에 Supabase 값이 주입되지 않아 `/admin`에서 "Supabase가 설정되지 않았습니다" 오류가 뜹니다.

3. **사이트가 안 열리거나 예전 버전이 보여요**  
   - Cloudflare Pages 배포 상태 확인.  
   - GitHub `main`에 최신 코드가 push되었는지, Actions에서 빌드가 성공했는지 확인.

4. **새 관리자 계정을 추가하고 싶어요**  
   - Supabase 대시보드 → Authentication → Users에서 이메일/비밀번호로 새 사용자 추가.  
   - 같은 Supabase 프로젝트를 쓰는 한, 그 계정으로 `/admin` 로그인 가능.
